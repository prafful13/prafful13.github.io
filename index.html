How I work

🔎

Build visibility before building a fix

When I hit a failure I don't understand, I instrument before I guess. Synthetic tests, distributed traces, dashboards first. The actual root cause of a problem is often only visible after you've built the right observability layer. Jumping straight to a fix is how you patch the wrong thing and carry the real issue forward.

📈

Own unowned critical problems

When a critical system has no owner and no documentation, I treat that as a signal to lean in, not step back. I've stepped into systems mid-failure with zero handoff, root-caused novel infrastructure problems from first principles, and redesigned the architecture from scratch when the original approach couldn't be salvaged.

🧠

Reason from constraints, not trends

Popular tools get rejected when the constraints don't fit. I've turned down Ansible, Chef, and agentless config management frameworks when runtime overhead, authentication constraints, or firewall rules made them the wrong call. The right tool is the one that fits your actual environment, not the one that looks good in a tech radar.

▶️

Use AI as an engineering multiplier

I use AI to accelerate analysis that would otherwise take days: finding patterns in large request datasets, identifying predictable burst windows that inform pre-warming strategies over reactive scaling. I also build and operate the infrastructure that AI coding agents run on, so I understand both sides of that boundary.

🎓

Roll out like a stakeholder, not a developer

Shipping a change to thousands of repos or across a multi-cluster production fleet is not the same as merging a PR. I work directly with the teams affected through adoption, run incremental rollouts with monitoring gates before the next step, and don't announce things as done until they're actually stable in production.

🔒

Security and ops are part of the design

I don't bolt security on at the end. Pull-based architectures that eliminate ad-hoc SSH access, isolation patterns that strengthen the security posture as a side effect of a reliability fix, zero secrets in git from day one on personal projects. These are design choices, not checklist items.

Projects

robinhood-pilot

Live ↗ GitHub ↗

An automated equity trading bot using Robinhood's Agentic MCP API, deployed on a local k3s cluster. Trades RSI mean-reversion signals on a configurable watchlist with four selectable strategies (RSI, MACD crossover, Bollinger Bands, RSI+MACD combo). All strategy and risk parameters are editable live from a Streamlit dashboard without a restart.

The architecture reflects how I approach production systems: zero secrets in git, a three-layer secrets model (macOS Keychain on the host, Bitnami Sealed Secrets for k8s, env vars and file mounts in pods), PostgreSQL on k3s for trade and portfolio history, and a full tax calculator with FIFO cost basis and short/long-term gain split built into the dashboard.
